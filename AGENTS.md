# Инструкции для разработчиков и ИИ-помощника

## Проект и домен
- **Название**: `boutique_production_sync_v2`
- **Описание**: Асинхронный сервис для загрузки и обработки прайсов для дальнейшего формирования предложений для клиентов. Обновления данных на сайтах и загрузка прайсов в Dropbox.
- **Основные сущности**: ...
- **Ограничения домена**:
  - Все операции синхронизации и обновления данных должны быть **идемпотентными**.
  - Строгая асинхронность: никаких блокирующих вызовов (sync I/O) в асинхронном контексте.
  - Обработка ошибок должна учитывать лимиты (rate limiting) и использовать ретраи с экспоненциальной задержкой (exponential backoff).

## Архитектура
- **Слоистая архитектура** (Clean Architecture / Hexagonal):
  - `src/api/` — роутеры, DTO, валидация входящих запросов.
  - `src/services/` — бизнес-логика, оркестрация процессов.
  - `src/repositories/` — абстракции для работы с БД и кэшем.
  - `src/integrations/` — клиенты внешних API (Битрикс24, сторонние сервисы).
- **Зависимости**: допустимы только сверху вниз (API → Services → Repositories/Integrations). Репозитории и интеграции не должны знать о сервисах.
- **Внедрение зависимостей (DI)**: использование `fastapi.Depends` и контейнеров зависимостей.
- **Работа с данными**: Репозитории работают через `SQLAlchemy 2.0` (async) и `redis.asyncio`.
- **Транзакционность**: управление сессиями БД через middleware или dependency injection (паттерн Unit of Work).
- **Кэширование**: кэширование данных реализуется на уровне сервисов или репозиториев с использованием Redis.

## Стандарты кодирования
- **Форматирование и линтинг**: `black`, `ruff` (настройки из `pyproject.toml` включая `ruff format` и проверку импортов), `mypy --strict`.
- **Типизация**: использование современных конструкций Python (PEP 604 `X | Y`, PEP 585 `list[X]`). Избегать `typing.Optional`, `typing.List` в пользу встроенных типов (все функции и методы аннотированы).
- **Нейминг**: осмысленные имена, отражающие бизнес-логику. Избегать аббревиатур, кроме устоявшихся (API, DTO, ID).
  - Переменные, функции, методы: `snake_case`
  - Классы: `PascalCase`
  - Константы: `UPPER_SNAKE_CASE`
  - Приватные методы/атрибуты: с префиксом `_` (один подчёркивание)
- **Логирование**: структурированное логирование через `python-json-logger`. Обязательные поля в каждом логе:
```json
  {
  "asctime": "2026-05-11T17:06:48",        // ISO8601, с временной зоной UTC
  "levelname": "INFO",                     // уровень: DEBUG, INFO, WARNING, ERROR, CRITICAL
  "name": "sync",                          // имя логгера (обычно __name__)
  "module_name": "/app/src/db/redis.py",   // полный путь к файлу (или относительный от корня проекта)
  "class_name": "RedisManager",            // имя класса (если применимо, иначе null)
  "method_name": "close",                  // имя функции/метода (если применимо, иначе null)
  "message": "Closing Redis connection..." // текст сообщения
}
  ```
  - Дополнительные поля (необязательные) могут быть добавлены через параметр extra в вызове логгера.
  - Конфигурация должна обеспечивать автоматическое добавление asctime, module_name, class_name, method_name. Для этого используется кастомный форматтер.

## Документация и комментарии
- **Докстринги** (тройные двойные кавычки) — **на русском**, содержат краткое описание, аргументы, возвращаемое значение, исключения.
- **Комментарии внутри кода** — **на русском**, поясняют сложную логику.
- **Логи и сообщения исключений** — **на английском** (для интернациональной поддержки и совместимости с внешними системами).

## Обработка ошибок

- **Базовое исключение**: все кастомные исключения наследуются от `BaseAppException` (расположен в `src/core/exceptions/base.py`).
- **Содержательность исключений**: каждый класс исключения должен определять:
  - `error_code` — строковой код ошибки (например, `"PRICE_NOT_FOUND"`),
  - `message` — человекочитаемое описание (на английском),
  - `status_code` — HTTP-статус (опционально),
  - `details` — дополнительные данные (опционально, dict).
- **Глобальный обработчик**: в `src/main.py` функция `configure_exception_handlers` регистрирует обработчик `BaseAppException`, который формирует стандартизированный JSON-ответ.
- **Формат ответа API при ошибке**:

```json
{
  "error_code": "PRICE_NOT_FOUND",
  "message": "Price by supplier 'lanset' not found.",
  "details": {"supplier": "lanset", "source": "price_api"},
  "request_id": "8f3e2d1c-..."
}
```
- **Не подавлять исключения без необходимости**: логировать с уровнем ERROR.

## Типовые задачи для ИИ

1. **Ревью предложений изменений в коде**
   - Проверять соответствие архитектуре, кодстайлу, типизации.
   - Указывать на потенциальные проблемы производительности или безопасности.
   - Ревью кода на предмет соответствия стандартам, поиска узких мест и нарушений асинхронности.

2. **Генерация unit-тестов**
   - Использовать `pytest` и `pytest-asyncio`.
   - Тесты должны покрывать бизнес-логику сервисов, репозитории тестировать с in-memory БД или моками.
   - Стремиться к покрытию ≥80%.

3. **Генерация кода**
   - Соблюдать все перечисленные стандарты.
   - Для новых эндпоинтов создавать Pydantic-схемы в `src/schemas/`.
   - Для новых таблиц — модель SQLAlchemy, alembic-миграцию и репозиторий.
   - Всегда добавлять тесты на новый функционал.

4. **Написание миграций (Alembic)**
   - Генерировать миграции через `alembic revision --autogenerate`.
   - Проверять, чтобы миграции были идемпотентными (обратимыми).

5. **Оптимизация SQL-запросов**
   - Устранение проблемы N+1, предложение индексов.

## Чего делать не нужно (Антипаттерны)
- **Не использовать синхронные библиотеки** (например, `requests`, `psycopg2`, `time.sleep`) внутри `async def` функций. Блокирующий код запрещён (используйте `asyncio.to_thread` при необходимости).
- **Не менять сигнатуры публичных API** (роутеров, интерфейсов сервисов) без явного запроса. Такие изменения требуют обновления документации и согласования.
- **Не переписывать работающий код «ради красоты»**, если это не несет практической пользы (производительность, читаемость, исправление багов).
- **Не использовать "голые" исключения** (`except Exception:` или `except:`) без логирования и проброса.
- **Не хардкодить конфигурации**, секреты и URL — использовать `pydantic-settings` и переменные окружения.
- **Не создавать циклические импорты** между слоями (например, репозиторий не должен импортировать сервисы).
- **Не игнорировать типизацию** — `# type: ignore` только в крайнем случае с комментарием причины.
